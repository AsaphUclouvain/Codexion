# Description

Codexion simulates a concurrent team of coders competing for shared dongles to compile, debug, and refactor their work. The goal is to implement a thread-safe scheduling and synchronization system that prevents deadlock and starvation, enforces cooldowns between resource uses, and detects burnout conditions when a coder waits too long without compiling.

# Instructions

## Compilation

From the project root:

```sh
make
```

This builds the executable `codexion` using `cc` with `-Wall -Werror -Wextra` and `-pthread`.

## Execution

### Command-line parameters

The program accepts either a single quoted string with 8 parameters or 8 separate parameters:

```sh
./codexion "<number_of_coders> <time_to_burnout> <time_to_compile> <time_to_debug> <time_to_refactor> <number_of_compiles_required> <dongle_cooldown> <scheduler>"
```

or:

```sh
./codexion <number_of_coders> <time_to_burnout> <time_to_compile> <time_to_debug> <time_to_refactor> <number_of_compiles_required> <dongle_cooldown> <scheduler>
```

### Example

```sh
./codexion 5 800 200 100 100 3 50 fifo
```

## Notes

- `scheduler` must be either `fifo` or `edf`.
- Time values are expressed in milliseconds and internally converted to microseconds.
- Invalid arguments print `Invalid Arguments` and return an error.

# Blocking cases handled

The implementation explicitly handles concurrency issues that can arise when multiple coder threads compete for shared dongles and monitor state.

- `Deadlock prevention`: coder threads acquire dongles in a consistent order by comparing left and right key indices, avoiding circular waits.
- `Starvation prevention`: coders use a priority queue per dongle, where scheduler choice (`fifo` or `edf`) determines access order and helps ensure all waiting coders progress.
- `Cooldown handling`: when a coder obtains a dongle, the code enforces a cooldown interval before the same dongle can be reused by another coder.
- `Precise burnout detection`: each coder thread tracks arrival time and uses `pthread_cond_timedwait()` against a calculated burnout deadline so the monitor can detect and react immediately if a coder waits too long.
- `Instant burnout alerting`: The output is protect by a flag named output_open. When a coder burns out, it sets that flag to `0` preventing another coder from printing again.
- `Log serialization`: all console output is protected by a single `output_mutex` so event messages remain ordered and free of interleaving.

# Thread synchronization mechanisms

The implementation uses POSIX threading primitives to coordinate access to shared resources and avoid race conditions.

## Mutexes (`pthread_mutex_t`)

- `output_mutex`: protects console logging and `output_opened`, ensuring only one thread prints at a time.
- `burnout_mutex`: protects shared monitor state such as `burnout_sig` and `finished_count`, enabling safe updates and condition signaling.
- `keys[i].mutex`: protects each dongle’s waiting queue and the enqueue/dequeue operations on that dongle.

Example: a coder locks `ctx->mutex` before inserting its request into the dongle queue and before waiting for the dongle condition.

## Condition variables (`pthread_cond_t`)

- `burnout_cond`: used to notify either the monitor thread or all coders when a burnout condition occurs or when coders finish normally.
- `keys[i].cond`: used to wake coders waiting for a specific dongle when the dongle becomes available or when a burnout occurs.

Example: when a coder drops a dongle, it broadcasts on `ctx->cond` so any waiting coder can re-evaluate availability.

## Thread-safe communication

- Coders request dongles via a per-key priority queue protected by `keys[i].mutex`.
- Waiting coders use `pthread_cond_timedwait()` while holding the same mutex, avoiding lost wakeups and guaranteeing the queue state stays consistent.
- Burnout detection is communicated through shared state protected by `burnout_mutex`, with `pthread_cond_broadcast()` to wake all waiting coders when one coder burns out.

## Race condition prevention

- Shared state updates are always guarded by the appropriate mutex.
- The `output_mutex` prevents mixed logging lines from concurrent printf calls.
- Dongle queue operations and wake-ups are serialized by `keys[i].mutex` and `keys[i].cond`, ensuring only the rightful next coder proceeds.
